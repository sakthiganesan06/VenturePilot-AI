# Deploying to Render — Guide

This guide details how to deploy the **AI Startup Blueprint Generator** to **Render** using the provided `render.yaml` configuration.

---

## Deployment Strategy: Docker Web Service

We use a **Docker-based deployment** for this application on Render. This is the most robust method because the application's RAG pipeline relies on `faiss-cpu` and `sentence-transformers`, which compile/install compiled C++ headers and python packages. Running within the Docker environment ensures all dependencies are built and run deterministically.

---

## Step-by-Step Deployment Instructions

### 1. Push Code to GitHub / GitLab
Make sure all your local changes (including `Dockerfile`, `requirements.txt`, and `render.yaml`) are committed and pushed to a remote git repository (e.g. GitHub):
```bash
git add .
git commit -m "chore: configure for Render deployment"
git push origin main
```

### 2. Connect to Render
1. Go to the [Render Dashboard](https://dashboard.render.com/) and sign in.
2. Click the **New** button in the top right corner and select **Blueprint**.
3. Connect your GitHub repository containing the project.
4. Render will auto-detect the `render.yaml` file in the repository.

### 3. Configure Service and Environment Variables
Render will read `render.yaml` and display a page to configure the deployment group. You will see a list of environment variables.
Provide values for the following variables under the **Environment Variables** section:

| Variable Name | Description | Example / Recommended Value |
| --- | --- | --- |
| `IBM_WATSONX_API_KEY` | Your watsonx.ai API key | `your-watsonx-api-key` |
| `IBM_WATSONX_PROJECT_ID` | Your watsonx.ai Project ID | `your-watsonx-project-id` |
| `IBM_COS_API_KEY` | Your IBM Cloud Object Storage API Key | `your-cos-api-key` |
| `IBM_COS_INSTANCE_CRN` | Your IBM COS Instance CRN | `crn:v1:bluemix:public:cloud-object-storage...` |

*Note: The following variables are already pre-filled in `render.yaml` but can be edited if necessary:*
- `IBM_WATSONX_URL`: `https://us-south.ml.cloud.ibm.com`
- `GRANITE_MODEL_ID`: `ibm/granite-3-3-8b-instruct`
- `IBM_COS_ENDPOINT`: `https://s3.us-south.cloud-object-storage.appdomain.cloud`
- `IBM_COS_BUCKET_KB`: `startup-knowledge-base`
- `IBM_COS_BUCKET_REPORTS`: `startup-reports`
- `IBM_COS_BUCKET_RESOURCES`: `startup-resources`
- `SECRET_KEY`: Automatically generated by Render on creation.
- `FLASK_DEBUG`: `false`

### 4. Create Resources
Click the **Apply** or **Create** button. Render will start spinning up the Docker container.

## Health Check and Verification

During the startup, Gunicorn is configured with a 120-second timeout. This allows the RAG pipeline enough time to download necessary model embeddings (e.g., SentenceTransformers) and setup index files on the container startup.

Once the service is active:
1. Render will provide a public URL (e.g., `https://startup-blueprint-generator.onrender.com`).
2. Verify the application health by appending `/api/health` to your app URL in the browser:
   `https://startup-blueprint-generator.onrender.com/api/health`
3. You should see a response like:
   ```json
   {
     "status": "ok",
     "granite": true,
     "rag": true,
     "cos": true
   }
   ```
4. Open the main URL in your browser to view the application UI.

## Troubleshooting & FAQ

### RAG Initialization Timeout
If you experience service health check timeouts during the initial deploy, check the logs on the Render dashboard. Download of the `sentence-transformers` embedding models from Hugging Face is performed when the application starts for the first time. The `Dockerfile` healthcheck and Gunicorn timeout have been configured to support this. If it still times out on Render's side, you may need to upgrade from the Free tier to a standard tier with more CPU/RAM resources.

### Local Directory Persistence
The application uses local folders `exports/`, `logs/`, and `.vector_db/` as temporary scratch space, but persists generated blueprints and PDFs to **IBM Cloud Object Storage (COS)**. This makes the service fully compatible with Render's ephemeral filesystem (which resets on every deploy or restart).
